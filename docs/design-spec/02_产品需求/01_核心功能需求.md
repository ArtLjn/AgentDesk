# 核心功能需求

## 1. 用户角色

| 角色 | 说明 |
| --- | --- |
| 普通用户 | 提交工单并查看处理结果 |
| 系统管理员 | 查看工单列表、统计数据、知识库和执行追踪 |
| 测试人员 | 构造不同类型工单验证系统流程 |
| 审核员 | 在审核工作台对待审核工单进行人工决策（v1.0 新增） |

当前系统已实现轻量登录鉴权：用户通过 `/api/auth/login` 登录，业务 API 由 session 保护；`auth_enabled=false` 时可在演示环境放行。审核员在毕设阶段仍通过 `reviewer_id` 字段标识，不实现企业级角色权限、审核员认领和多人会签。

## 2. 功能清单

### 2.1 工单提交

用户可以输入工单内容并提交。系统应返回工单 ID 和初始状态。

验收标准：

- 内容为空时应拒绝或提示。
- 后端应先调用 TicketIntentAgent 提取标题、分类、优先级、影响范围、联系方式等字段。
- 提交成功后返回 `ticket_id`。
- 工单初始状态为 `received`。
- 后台自动触发工作流。

### 2.2 批量工单提交

系统支持一次提交多条工单，用于测试并发处理能力。

验收标准：

- 单次提交数量应限制在合理范围内。
- 每条工单独立生成 ID。
- 单条失败不影响其他工单初始化。

### 2.3 智能分类与优先级识别

系统应自动识别工单分类和优先级。

分类范围：

- `technical`：技术问题。
- `billing`：账务问题。
- `complaint`：投诉。
- `inquiry`：咨询。

优先级范围：

- `P0`：紧急。
- `P1`：高。
- `P2`：中。
- `P3`：低。

### 2.4 自动路由

系统应根据分类和优先级选择后续流程：

- 投诉类进入升级处理。
- P0 工单进入升级处理。
- 其他工单进入处理与审核流程。

### 2.5 工单处理

处理 Agent 应根据工单内容、分类、优先级和知识库上下文生成处理方案。

验收标准：

- 处理结果不能为空。
- 处理结果应写入工单详情。
- 处理失败时应记录错误或进入降级逻辑。

### 2.6 质量审核

审核 Agent 应对处理结果进行评分。评分达标后进入通知流程，不达标则重试。

验收标准：

- 审核评分范围为 0 到 1。
- 默认通过阈值为 0.7。
- 重试次数不超过配置上限。
- 达到重试上限后应转入人工审核，而不是直接把低质量结果返回给用户。

### 2.7 知识库管理

管理员可以上传知识文档，供处理 Agent 检索使用。

验收标准：

- 上传时必须包含标题和正文。
- 上传成功后返回分块数量。
- 知识库不可用时接口返回明确错误。

### 2.8 实时进度推送

前端应能实时获取工单处理状态。

验收标准：

- 支持订阅单个工单进度。
- 支持全局监控所有工单进度。
- 每个节点完成后推送状态更新。

### 2.9 执行追踪

系统应能查询某个工单的执行轨迹。

验收标准：

- 可根据工单 ID 查询 trace。
- 可查看 span 树。
- 可查看节点耗时和执行状态。

### 2.10 统计分析

系统应提供基础统计数据。

验收标准：

- 支持分类分布统计。
- 支持优先级分布统计。
- 支持完成率、失败率、平均重试次数等指标。
- 支持满意度反馈统计。

### 2.11 登录鉴权

系统应提供后台管理端基础登录能力。

验收标准：

- 支持用户名密码登录。
- 登录成功后写入 session cookie。
- 未登录访问业务 API 时返回 401。
- 支持查询当前登录状态。
- 支持退出登录。
- 设置页和 API 不暴露真实密钥内容。

### 2.12 人工审核

对于 AI 处理路径无法可靠闭环的工单，系统应转入人工审核流程。详细设计见 [01_正式设计/09_人工审核工作台设计.md](../01_正式设计/09_人工审核工作台设计.md)。

**触发场景**：

| 场景 | 触发条件 | 默认推荐决策 |
| --- | --- | --- |
| 投诉 / P0 升级 | 路由判断为 complaint 或 P0 | reprocess 或 reject |
| AI 审核失败兜底 | retry_count ≥ 3 且 review_score < 阈值 | rewrite 或 reject |
| 处理异常 | 工作流执行抛出未恢复异常 | reprocess |
| 用户主动申请 | 工单完成后用户反馈不满意 | reprocess |

**功能验收标准**：

- 工单进入 `human_review_wait` 节点时状态变为 `pending_human_review`。
- `human_reviews` 表中插入一行 pending 记录，关联 CoordinatorAgent 生成的辅助决策建议。
- WebSocket 广播 `review_requested` 事件。
- 审核工作台支持按 trigger_type / category / priority 筛选待审核队列。
- 审核员可提交 4 种决策：approve（通过）/ rewrite（改写）/ reprocess（重处理）/ reject（驳回）。
- 审核员可提交 `request_info` 决策，请求用户补充订单号、支付流水号等关键信息。
- decision=rewrite 时必须填写 rewritten_result。
- 提交决策后工作流自动恢复，按决策结果路由到 notify / process / complete。
- decision=request_info 后工单状态变为 `waiting_user_input`，暂不恢复工作流。
- WebSocket 广播 `review_decided` 事件。
- 审核统计接口返回 `ai_adoption_rate`（AI 建议采纳率）指标。

**非功能验收标准**：

- 同一工单同时只能有一个 pending 审核记录，重复提交决策返回 409。
- 决策记录不可修改，但可触发新的 `user_request` 审核。
- 审核过程被 trace 系统完整记录，包含 `human_decision` span。

### 2.13 用户补充信息

当工单需要进一步业务处理但缺少必要字段时，系统应支持审核员向用户发起补充请求，用户补充后自动恢复 Agent 处理流程。

典型场景：

| 场景 | 需要补充的信息 |
| --- | --- |
| 退款未到账 | 订单号、支付流水号 |
| 重复扣费 | 支付凭证、扣费时间 |
| 投诉服务处理 | 联系方式、具体事件时间 |
| 账户异常核查 | 用户 ID、异常截图或错误码 |

功能验收标准：

- 审核员在审核工作台点击“请求补充”后，工单进入 `waiting_user_input`。
- `human_reviews` 当前审核记录被更新为 `decision=request_info`、`status=decided`。
- 系统写入一条 `ticket_messages` 审核员消息，内容为补充说明。
- 工单详情页在 `waiting_user_input` 状态下显示用户补充输入框。
- 用户提交补充信息后，系统写入 `ticket_messages` 用户消息。
- 系统读取沟通记录构造 `conversation_context`，从 `process` 节点恢复处理。
- 补充后的工单仍需经过 ReviewerAgent 质量审核，不能绕过 `review` 节点直接完成。
- 非 `waiting_user_input` 状态提交补充消息应返回 409。
