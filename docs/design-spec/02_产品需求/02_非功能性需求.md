# 非功能性需求

## 1. 可用性

系统应支持本地开发环境稳定运行。即使 LLM 或 Qdrant 暂时不可用，核心工单流程也应通过降级逻辑继续运行。

要求：

- 后端启动失败时应输出明确日志。
- Qdrant 初始化失败不影响 API 主服务启动。
- Agent 调用失败时应进入重试或降级逻辑。

## 2. 响应性能

系统面向演示和本科毕设，不要求高并发生产性能，但应满足基本交互体验。

要求：

- 工单提交接口应快速返回，不等待完整工作流结束。
- 后台处理过程通过 WebSocket 更新。
- 列表查询、详情查询和审核队列查询应在演示环境中快速响应。

## 3. 可维护性

代码结构应保持清晰，便于论文说明和后续修改。

要求：

- API、Agent、Workflow、Tool、Core 分层清晰。
- 新增功能优先放入已有模块边界内。
- 避免在单个文件中堆积过多职责。
- 文档与代码关键流程保持同步。

## 4. 可测试性

系统应具备自动化测试基础。

要求：

- 覆盖工作流状态流转。
- 覆盖核心 Agent 降级逻辑。
- 覆盖主要 API 接口。
- 测试不依赖真实外部 LLM 服务。

## 5. 可解释性

多 Agent 系统需要能解释“为什么得出这个处理结果”。

要求：

- 工单详情应展示分类、优先级、处理结果和审核评分。
- trace 接口应展示节点执行顺序和耗时。
- trace 决策接口应展示关键路由、质量门禁和重试边界判断。
- WebSocket 消息应包含当前节点和状态。

## 6. 安全性

本科毕设阶段实现基础安全约束，不扩展企业级权限系统。

要求：

- 业务 API 默认要求登录。
- 密码使用哈希保存和校验，不在文档或仓库保存明文密码。
- 不提交 `.env`、密钥、凭证等敏感信息。
- API 不在日志中打印完整密钥。
- 上传知识库内容仅作为演示，不处理敏感真实数据。
- 前端设置页不应暴露真实生产凭证。

## 7. 部署性

系统应支持本地运行和 Docker Compose 演示。

要求：

- 后端可通过 `uvicorn` 启动。
- 前端可通过 `npm run dev` 启动。
- Qdrant 等依赖可通过 Docker Compose 启动。
- MySQL 连接通过 `config.yaml` 或环境变量配置，测试库和生产库应隔离。
- README 或文档中说明启动方式。
