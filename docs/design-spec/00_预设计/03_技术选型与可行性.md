# 技术选型与可行性

## 1. 技术栈总览

| 类别 | 技术 | 选择原因 |
| --- | --- | --- |
| 后端框架 | FastAPI | 支持异步接口、参数校验和自动生成 API 文档 |
| 工作流编排 | LangGraph | 适合表达多 Agent 状态机和条件路由 |
| 大模型接口 | OpenAI SDK 兼容接口 | 便于接入 Ollama、DeepSeek 或其他兼容服务 |
| 数据模型 | Pydantic | 与 FastAPI 结合紧密，适合请求和响应校验 |
| 数据库 | SQLite | 部署简单，适合本科毕设和本地演示 |
| 向量数据库 | Qdrant | 支持语义检索，可用于 RAG 演示 |
| 前端框架 | React + TypeScript | 适合构建管理端界面，类型约束清晰 |
| UI 样式 | TailwindCSS + shadcn/ui | 快速构建一致的后台管理界面 |
| 测试框架 | pytest | Python 项目常用测试框架 |
| 容器化 | Docker Compose | 方便启动 API、前端、Qdrant 等服务 |

## 2. FastAPI 可行性

FastAPI 能够满足系统接口层需求：

- 支持异步接口，适合后台执行工单工作流。
- 支持 Pydantic 模型校验，减少手写参数检查。
- 支持 WebSocket，可实现工单状态实时推送。
- 自动生成 Swagger 文档，便于联调和论文截图展示。

当前项目已经在 `src/multi_agent_system/api/app.py` 和 `src/multi_agent_system/api/routes.py` 中完成基础实现。

## 3. LangGraph 可行性

工单处理流程天然具有状态机特征：不同分类和优先级会进入不同路径，审核失败后可能重试，失败后进入兜底处理。LangGraph 适合表达这种有向图流程。

相比在普通函数里堆叠 `if/else`，LangGraph 的优势是：

- 流程节点清晰，便于画图和解释。
- 条件路由可独立维护。
- 后续扩展新节点较方便。
- 适合体现多 Agent 协同思想。

## 4. SQLite 可行性

本科毕设阶段，系统主要面向功能演示和核心流程验证，不需要高并发数据库能力。SQLite 具有以下优势：

- 无需独立数据库服务，降低部署复杂度。
- 支持工单、用户、检查点、trace、span 等数据存储。
- 与 `aiosqlite` 结合后能满足异步接口调用。
- 数据表结构简单，便于论文中展示。

如果后续需要扩展到生产系统，可将 SQLite 替换为 MySQL 或 PostgreSQL。

## 5. Qdrant 可行性

知识库检索是本项目的重要亮点，但不是所有演示流程都依赖知识库。Qdrant 作为可选组件使用：

- 可上传工单处理知识文档。
- 可将文档分块后写入向量库。
- 处理 Agent 在生成方案前检索相关知识片段。
- 当 Qdrant 不可用时，系统核心工单流程仍可运行。

这种设计既能展示 RAG 能力，也避免系统对外部服务过度依赖。

## 6. 前端技术可行性

React + TypeScript 适合实现本项目的后台管理端：

- 工单列表、详情、统计和设置页面都是典型管理端页面。
- TypeScript 类型定义可与后端响应结构保持一致。
- WebSocket hook 可用于实时刷新状态。
- shadcn/ui 组件能减少基础 UI 开发成本。

当前前端页面包括 `Dashboard`、`Tickets`、`TicketDetail`、`Knowledge`、`AgentMonitor` 和 `Settings`，已经覆盖毕业设计演示所需界面。

## 7. 风险与应对

| 风险 | 影响 | 应对策略 |
| --- | --- | --- |
| LLM 服务不可用 | Agent 无法生成高质量结果 | 保留关键词分类、占位处理和降级逻辑 |
| Qdrant 未启动 | RAG 检索不可用 | 将知识库设为可选，核心流程继续运行 |
| 工作流异步执行难调试 | 前端状态不一致 | 使用 WebSocket 和 trace/span 记录执行过程 |
| 文档范围过大 | 毕设难以收尾 | 明确排除企业级权限、SLA、人工客服台等复杂能力 |
| 测试环境差异 | 演示不稳定 | 保留 pytest 自动化测试和本地 Docker 启动方式 |
