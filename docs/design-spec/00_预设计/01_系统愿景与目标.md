# 系统愿景与目标

## 1. 项目背景

在客服、运维和企业内部服务场景中，工单通常需要经历问题识别、分类分派、处理建议生成、质量审核和结果通知等步骤。传统工单系统依赖人工分类和处理，效率受人员经验影响较大，也难以及时沉淀历史知识。

随着大语言模型和 Agent 技术的发展，可以将工单处理拆解为多个相对独立的智能任务：分类 Agent 负责识别问题类型，处理 Agent 负责结合知识库生成解决方案，审核 Agent 负责评估结果质量，协调 Agent 负责异常和升级处理。本项目以此为切入点，设计并实现一个基于多智能体协同的智能工单处理系统。

## 2. 系统愿景

系统希望实现一个轻量但完整的智能工单处理闭环：用户提交工单后，系统能够自动识别工单类型和优先级，根据路由规则调用不同 Agent 完成处理，并在前端实时展示处理状态、处理结果、审核评分和执行轨迹。

本项目强调“可解释的多智能体协作”，不仅要给出最终结果，还要能展示工单经过了哪些节点、每个节点产生了什么中间信息，以及系统为何进行自动回复、升级或重试。

## 3. 建设目标

### 3.1 功能目标

- 支持用户提交单条或批量工单。
- 支持工单分类，包括技术问题、账务问题、投诉和咨询。
- 支持优先级识别，包括 P0、P1、P2、P3。
- 支持咨询类工单自动回复。
- 支持投诉类或高优先级工单升级处理。
- 支持一般工单由处理 Agent 生成解决方案。
- 支持审核 Agent 对处理结果进行质量评分。
- 支持审核不通过时自动重试，超过上限后标记失败。
- 支持知识库文档上传与检索。
- 支持前端查看工单列表、详情、执行追踪和统计数据。

### 3.2 技术目标

- 使用 LangGraph 表达多 Agent 工作流，体现清晰的状态流转和条件路由。
- 使用 FastAPI 提供 REST API 和 WebSocket 实时推送能力。
- 使用 React + TypeScript 实现基础管理端页面。
- 使用 SQLite 保存工单、用户、检查点、trace 和 span 数据。
- 使用 Qdrant 作为可选向量数据库，实现知识库检索增强。
- 使用重试、降级、缓存和模型路由提升系统稳定性。

### 3.3 论文目标

论文重点围绕以下问题展开：

- 多 Agent 如何围绕工单生命周期进行职责拆分。
- LangGraph 如何表达工单状态机和条件路由。
- RAG 如何提升处理 Agent 的解决方案生成能力。
- 审核与重试机制如何提升输出结果的稳定性。
- 执行追踪如何提高多 Agent 系统的可解释性。

## 4. 项目边界

### 4.1 纳入范围

- 工单提交、查询、列表、状态更新。
- 多 Agent 自动分类、处理、审核和协调。
- 知识库文档上传与检索增强。
- Trace 执行轨迹记录与前端展示。
- 基础统计分析与评价指标。
- 本地或 Docker 环境下的演示部署。

### 4.2 不纳入范围

- 完整客服 IM 系统。
- 多租户权限和组织架构管理。
- 企业级 SLA、排班和人工客服工作台。
- 复杂审批流、财务系统对接和真实短信邮件发送。
- 大规模分布式部署和高并发压测。
- 生产级安全审计和细粒度权限控制。

## 5. 预期成果

- 一套可运行的智能工单处理系统。
- 一套前后端联动的演示页面。
- 一组覆盖核心流程的测试用例。
- 一份系统设计文档和接口说明。
- 一篇能支撑答辩的毕业论文，重点体现系统设计、实现和测试过程。
