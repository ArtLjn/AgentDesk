# 毕设范围控制

## 1. 范围控制原则

本项目来自学习项目，已有内容较多。毕业设计应聚焦“智能工单处理系统”，避免把基础 Agent、论文阅读助手、个人知识库等学习模块全部纳入论文主线。

范围控制原则：

- 只讲与工单系统直接相关的模块。
- 重点突出多智能体协同、工作流编排和执行追踪。
- 保留 RAG 作为亮点，但不扩展成完整知识管理平台。
- 前端只作为管理端演示，不扩展成完整客服系统。

## 2. 论文主线建议

论文标题：基于多智能体协同的智能工单处理系统设计与实现。

建议章节主线：

1. 绪论：说明工单处理背景、多智能体和大模型应用价值。
2. 相关技术：介绍 FastAPI、LangGraph、RAG、React、SQLAlchemy async、MySQL。
3. 需求分析：说明角色、功能需求、非功能需求。
4. 系统设计：说明总体架构、多 Agent 协同、数据库、接口。
5. 系统实现：说明分类、处理、审核、工作流、前端页面实现。
6. 系统测试：说明测试环境、测试用例和结果。
7. 总结与展望：总结成果，展望权限、人工客服、SLA 等扩展。

## 3. 应避免的过度设计

| 不建议扩展 | 原因 |
| --- | --- |
| 多租户权限系统 | 工作量大，偏离多 Agent 主线 |
| 完整客服聊天系统 | 会变成 IM 项目 |
| 复杂 SLA 和排班 | 业务复杂度过高 |
| 真实支付、退款、短信邮件对接 | 外部依赖多，不适合毕设演示 |
| 大规模分布式部署 | 与本科毕设重点不匹配 |
| 自训练模型或微调 | 成本和风险较高 |

## 4. 可作为展望的内容

以下内容适合作为论文展望，而不是当前必须实现：

- **人工审核工作台的深度扩展**（审核员账户体系、技能组分配、SLA 超时、认领锁定、多人会签、工单转派链路、真实邮件/短信通知审核员）。注：v1.0 已实现轻量人工审核闭环，详见 [01_正式设计/09_人工审核工作台设计.md](../01_正式设计/09_人工审核工作台设计.md) 第 1.3 节范围边界。
- 支持多租户和权限控制。
- 引入更完整的 SLA 规则。
- 使用更强的向量检索和重排序。
- 基于历史处理结果形成持续学习闭环。
- 数据库迁移体系、备份恢复和读写分离等生产级数据库治理。
- 引入更细粒度的安全审计。

## 5. 最小可交付版本

最小可交付版本应包含：

- 后端 API 可启动。
- 前端页面可访问。
- 用户可提交工单。
- 工单可被自动分类和处理。
- 至少两条不同路由路径可演示。
- 工单详情能查看处理结果。
- 测试用例能覆盖核心流程。
- 文档能说明系统设计。

## 6. v1.0 范围扩展说明

v1.0 版本把"人工审核工作台"从纯展望调整为**部分纳入范围**。原因：

- 原 escalate 节点声称"升级到人工"但实际只写文字消息，存在答辩逻辑漏洞。
- AI 审核失败重试超限后直接归档，缺少合理兜底，影响论文严谨性。

纳入部分（详见 [01_正式设计/09_人工审核工作台设计.md](../01_正式设计/09_人工审核工作台设计.md)）：

- 审核队列与决策 API。
- 工作流暂停 / 恢复机制（基于持久化状态，未引入 checkpointer）。
- 审核工作台前端页面。
- CoordinatorAgent 辅助决策建议。
- `human_reviews` 表持久化。
- 轻量登录鉴权。

仍为展望的部分：

- 审核员角色权限、认领锁定和操作审计。
- 技能组分配与排班。
- SLA 超时自动转派。
- 多人会签 / 工单转派链路。
- 真实邮件 / 短信通知审核员。
